# B站视频与评论采集框架项目报告

## 1. 项目概况

| 项目 | 内容 |
|---|---|
| 当前版本 | v0.6.0 |
| 基线归档 | Git 标签 `v0.1.0` |
| 更新日期 | 2026-06-19 |
| 开发语言 | Python 3.12 |
| 数据存储 | SQLite schema v3 |
| 数据源 | B站匿名页面内部 API |
| 输出 | CSV、JSONL、Parquet、CLI inspect、批次摘要、离线 HTML 生命周期与内容报告 |

v0.2 将原有一级评论脚本升级为可扩展的数据采集框架；v0.3 增加可恢复的单进程批处理；v0.4 增加离线数据质量报告；v0.5 增加短期趋势与关键词分析；v0.6 将分析核心调整为评论生命周期与条件式讨论迁移。

项目保持匿名、串行、低频原则，不接入 OAuth，不保存 Cookie，不处理验证码，也不实施反风控规避。

## 2. 当前能力

- 获取视频标题、简介、封面、分区、时间、时长、版权和状态；
- 保存分P的 `cid`、标题、时长和画面尺寸；
- 保存播放、弹幕、评论、收藏、投币、分享、点赞和排名快照；
- 按热门或时间顺序遍历一级评论；
- 可选完整遍历楼中楼，保存根评论和父评论关系；
- 保存置顶类型、评论状态及最小作者字段；
- 评论当前值和每次运行观测分离；
- 一级评论和楼中楼均支持事务化检查点；
- 支持从 v0.1 数据库自动迁移；
- 导出评论、视频指标历史和评论指标历史；
- 使用 `inspect` 查看本地状态。
- 从 CSV 固化批量任务，失败后继续并支持按批次恢复；
- 保存批次及任务状态，输出不含评论正文的 JSON 摘要；
- 可选导出 Parquet；
- 生成不含评论正文和用户标识的单文件离线 HTML 报告；
- 可选生成一级评论词频、TF-IDF 和关键词共现统计；
- 生成一级评论发布后生命周期、分位时间、长尾和多视频共同窗口比较；
- 楼中楼数据充分时分析讨论迁移、首次回复延迟和线程集中度；
- 采用 MIT 许可证。

## 3. 架构

```text
CLI
 │
 ├─ crawl ─► 抓取服务 ─► SourceAdapter ─► BilibiliWebSource
 │                │                              │
 │                └────────► SQLite ◄────────────┘
 │
├─ export ─────────────────► SQLite ─► CSV / JSONL / Parquet
├─ inspect ────────────────► SQLite ─► JSON 状态
├─ report ─► 生命周期 / 讨论迁移 / 可选内容分析 ─► 离线 HTML
└─ batch ─► CSV 快照 ─► 单进程任务队列 ─► JSON 摘要
```

| 模块 | 职责 |
|---|---|
| `models.py` | 类型化视频、分P、指标、评论和分页模型 |
| `source.py` | 数据源适配器协议 |
| `client.py` | B站内部 API、WBI、字段映射、限速和重试 |
| `service.py` | 根评论、楼中楼和恢复流程编排 |
| `database.py` | schema v3、迁移、事务、当前记录、观测和批次状态 |
| `exporter.py` | 多实体 CSV/JSONL/Parquet 导出 |
| `report.py` | 聚合统计、趋势门槛和单文件 HTML 生成 |
| `content.py` | 一级评论规范化、分词、TF-IDF 和共现统计 |
| `lifecycle.py` | 生命周期、共同窗口、讨论迁移和集中度指标 |
| `cli.py` | `crawl`、`export`、`inspect`、`report`、`batch` |
| `batch.py` | CSV 校验、批次执行、恢复和摘要 |

业务层不再处理原始响应字典。B站接口路径和字段差异集中在适配器内，脱敏 JSON 夹具用于检测响应契约变化。

## 4. 数据模型

### 4.1 当前状态表

- `videos`：视频标识、作者、简介、封面、分区、发布时间、时长和状态。
- `video_pages`：分P及画面信息。
- `comments`：评论正文、当前指标、排序、父子关系、置顶和最小作者字段。

### 4.2 历史观测表

- `video_observations`：每次运行的视频互动指标。
- `comment_observations`：每次运行的评论点赞、回复数、排序位置和状态。

同一评论只保留一条主记录，但不同运行保留独立观测。数据缺失不等于删除，因此系统不自动推断删除状态。

### 4.3 运行与检查点

`crawl_runs` 保存数据源、排序方式、回复范围、根评论游标、楼中楼根评论和页码、运行完整性及结束原因。

写入一页评论与推进相应检查点在同一事务中完成。热门根游标超过 6 小时会被判定过期，旧运行以 `stale_cursor` 结束，新运行从头开始。

### 4.4 数据库迁移

使用 `PRAGMA user_version=3` 标识当前 schema。升级按 `v1 → v2 → v3` 顺序执行：

1. 在事务中添加视频、运行和评论字段；
2. 创建分P与观测表；
3. 将旧 `hot_rank` 回填为 `sort_rank`；
4. 保留已有视频、评论和运行数据；
5. 创建 `batch_runs` 和 `batch_items`；
6. 设置 schema 版本。

迁移可重复检查，不会重复添加字段或删除旧记录。

### 4.5 批次表

- `batch_runs`：清单路径和 SHA-256、状态、任务统计、时间和摘要路径。
- `batch_items`：CSV 行号、固化目标、抓取模式、状态、关联抓取运行和错误。

恢复批次时跳过成功项，将失败或中断项恢复为待执行。恢复使用数据库快照，因此原 CSV 修改不会改变既有批次。

## 5. 抓取策略

- `hot` 对应页面热门排序，默认保持 v0.1 行为。
- `time` 对应时间排序，适合更稳定地遍历新增内容。
- `root` 只请求一级评论。
- `all` 在一级评论完成后，根据回复数逐个分页请求楼中楼。
- 所有请求串行，默认间隔 1 秒。
- 传输错误、429 和 5xx 最多重试 3 次。
- 优先遵循数值或 HTTP 日期形式的 `Retry-After`；否则使用指数退避和随机抖动。
- 请求日志只记录路径、状态、次数和耗时，不记录评论正文或完整查询参数。
- 一个批次共享同一客户端和请求节奏；不增加批次级自动重试。

## 6. 隐私和合规

数据库不保存头像、签名、性别、VIP、粉丝牌或用户主页扩展资料。项目也不默认保存完整原始响应。

以下能力明确不实现：验证码处理、代理池、设备伪装、Cookie 托管、封禁规避、高并发和用户画像。

B站评论端点属于页面内部接口，不是正式开放 API。公开可见数据仍应按照平台规则、适用法律和最小化原则处理；真实数据库、日志和导出文件不应提交到公开 GitHub 仓库。

## 7. 测试与质量保障

测试覆盖：

- 视频、一级评论和楼中楼脱敏响应契约；
- 热门/时间排序参数及 WBI 签名；
- 匿名 WBI 密钥、登录和风控错误；
- `Retry-After`、指数退避和服务端重试；
- 多页抓取、根/子评论关系和独立检查点；
- 失败续抓及热门游标过期重启；
- v0.1→v2 无损迁移和重复打开；
- 视频与评论历史观测；
- 数据隔离、隐私字段约束及 CSV/JSONL/Parquet；
- CLI `inspect`。
- CSV 清单校验、批次继续执行、退出码、摘要和恢复；
- v2→v3 迁移及中断项目恢复；
- 三类实体的可选 Parquet 导出。

默认测试全部离线。CI 使用 Python 3.12 运行安装、Pytest 和 `compileall`。在线 canary 仅用于人工验证，不进入普通 CI。

离线报告测试覆盖 HTML 转义、生命周期桶、分位时间、多视频共同窗口、讨论迁移门槛、线程集中度、7天辅助趋势、私密字段排除和CLI退出码。内容分析继续覆盖清洗、TF-IDF、共现和楼中楼排除。

## 8. 已知限制

1. 内部接口路径、WBI 和响应字段可能变化。
2. 热门排序在跨页期间可能变化，主键可去重但无法恢复接口未返回的数据。
3. 匿名访问可能因地区、内容状态或风控返回不完整结果。
4. 当前仅接受 BV 号或包含 BV 号的 URL。
5. 不采集弹幕、字幕正文、推荐关系和用户主页。
6. 当前无内置定时调度或 Web UI；批量任务由单进程队列执行。
7. 楼中楼会显著增加请求数量，应继续保持低频使用。
8. 离线报告不会对少于 3 个观测点或不足 24 小时的数据作趋势判断，超过 36 小时的观测缺口不做插值。
9. 关键词是基于当前一级评论的统计结果，不包含情绪判断，也不能代表全部受众意见。
10. 讨论迁移只在完整楼中楼运行且覆盖率至少90%时生成；点赞数只是高互动代理，不能证明评论质量或用户偏好的因果关系。

## 9. 后续路线

- 后续分析：生命周期阶段关键词、讨论树和情绪分析保持独立可选。
- 独立模块：标签、字幕元数据、弹幕和官方 OAuth 数据源。

在已有历史数据足以支持趋势分析前，不建议优先建设 Web UI。
