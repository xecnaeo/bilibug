# B站一级评论采集器项目报告

## 1. 项目概况

| 项目 | 内容 |
|---|---|
| 项目名称 | B站一级评论采集器 |
| 归档版本 | v0.1.0 |
| 归档日期 | 2026-06-18 |
| 开发语言 | Python 3.12 |
| 使用形态 | 命令行工具 |
| 数据存储 | SQLite |
| 导出格式 | CSV、JSONL |
| 登录策略 | 匿名访问，不保存账号或 Cookie |

本项目用于低频采集 B站公开视频的一级评论。用户可以输入一个或多个 BV 号或视频 URL，程序按热门顺序遍历评论分页，将视频、评论和抓取运行状态持久化到 SQLite，并将已有数据导出到通用分析格式。

当前版本定位为可运行、可测试的最小版本。它不包含楼中楼、定时任务、Web 界面、用户画像或反风控功能。

## 2. 目标与验收状态

### 2.1 已实现目标

- 支持 BV 号和包含 BV 号的视频 URL。
- 支持依次抓取单个或少量视频。
- 使用页面内部 WBI 评论 JSON 接口获取热门排序的一级评论。
- 使用低频串行请求，避免主动制造高并发流量。
- 对超时、HTTP 429 和服务端错误执行有限重试。
- 明确识别登录要求、HTTP 401/403、验证码和风控响应，不尝试绕过。
- 使用 SQLite 保存视频、评论和抓取游标。
- 评论按视频和评论 ID 幂等更新，重复运行不会产生重复记录。
- 分页数据和下一页游标在同一数据库事务中提交，可在中断后续抓。
- 支持 UTF-8 BOM CSV 和 UTF-8 JSONL 导出。
- 默认测试完全离线，不依赖 B站服务可用性。

### 2.2 完整性的定义

“抓取完成”表示程序从第一页开始或从已保存游标恢复，持续请求至接口返回分页结束。由于热门排序会动态变化，且匿名接口可能隐藏已删除、审核中、折叠或受权限限制的评论，本项目不承诺历史意义上的绝对完整性。

## 3. 系统设计

### 3.1 数据流

```text
BV 号或视频 URL
       │
       ▼
  解析目标 BV 号
       │
       ▼
获取视频元数据 ──► 写入/更新 videos
       │
       ▼
获取匿名 WBI 密钥并签名
       │
       ▼
按热门顺序请求评论页
       │
       ├──► 写入/更新 comments
       └──► 原子更新 crawl_runs 游标
                    │
                    ▼
              CSV / JSONL 导出
```

### 3.2 模块职责

| 模块 | 职责 |
|---|---|
| `cli.py` | 定义 `crawl`、`export` 命令及退出码 |
| `service.py` | 解析 BV 号，编排视频与分页抓取 |
| `client.py` | HTTP 限速、重试、接口调用、响应校验和错误映射 |
| `wbi.py` | 提取 WBI 密钥并生成签名参数 |
| `database.py` | 建表、事务、幂等更新和断点状态 |
| `exporter.py` | 固定字段顺序并输出 CSV/JSONL |
| `errors.py` | 对用户可理解的领域错误进行分类 |

所有易变化的 B站内部接口细节集中在 `client.py` 和 `wbi.py`，避免接口变化扩散到数据库和 CLI。

## 4. 数据模型

### 4.1 `videos`

保存 BV/AV 标识、标题、UP 主公开 ID 和昵称、原始输入及首次/最近抓取时间。`bvid` 为主键，`aid` 唯一。

### 4.2 `comments`

以 `(bvid, rpid)` 为复合主键，保存：

- 评论 ID、正文和创建时间；
- 点赞数、回复数和本次热门位置；
- 作者公开 ID、昵称和等级；
- 首次发现时间和最近发现时间。

重复抓取会刷新动态字段，但不会因为评论本次未出现就推断其已被删除。楼中楼只可能存在于接口原始响应中，不会被请求、展开或持久化。

### 4.3 `crawl_runs`

保存每次抓取的开始/完成时间、运行状态、下一页游标、已遍历数量和错误。评论页写入与游标推进在同一事务中完成：页面写入失败时不会错误推进游标；进程异常退出后，状态为 `running` 的运行也可继续使用已提交游标。

## 5. CLI 与运行方式

安装：

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -e .
```

抓取：

```powershell
bili-comments crawl BV1xx411c7mD
bili-comments --db result.db crawl BV1xx411c7mD BV1yy411c7mE
```

导出：

```powershell
bili-comments --db result.db export BV1xx411c7mD --format csv --output comments.csv
bili-comments --db result.db export BV1xx411c7mD --format jsonl --output comments.jsonl
```

默认数据库为 `data/comments.db`。`data/` 已加入 `.gitignore`，避免将真实采集数据误提交到代码仓库。

## 6. 可靠性与失败处理

- 所有请求串行执行，默认请求间隔为 1 秒。
- 超时、传输错误、HTTP 429 和 5xx 最多重试 3 次，采用指数退避。
- HTTP 401/403 和 API 未登录响应映射为登录或匿名访问受限。
- HTTP 412、API `-352` 和 `-412` 映射为风控或验证码错误。
- 无下一页游标但仍声称存在后续数据时，程序停止并报告结构错误，防止无限循环。
- 数据库使用复合主键去重；重复抓取更新当前快照，不删除旧记录。
- CSV 使用 UTF-8 BOM，便于常见中文表格软件识别；正文换行和引号由标准 CSV 模块转义。

## 8. 安全、隐私与合规边界

- 工具只使用匿名访问，不接收或保存账号密码、Cookie、访问令牌。
- 只保存接口公开返回的最小作者字段，不额外访问用户主页或下载头像。
- 不实现验证码处理、代理池、设备指纹伪造、并发绕限或封禁规避。
- 公开可见不等于可以无限制收集或传播。使用者应遵守平台规则、适用法律和个人信息最小化原则。
- 商业化、大规模采集、长期用户跟踪、用户画像或对外发布数据前，应单独进行授权和法律评估。
- SQLite 数据库和导出文件可能包含个人信息，不应提交到公开 GitHub 仓库。

## 9. 已知限制与维护风险

1. 评论端点属于页面内部接口，不是稳定性受保证的正式开放 API。
2. WBI 参数、接口路径、返回字段或匿名访问策略变化时，需要更新客户端和固定响应测试。
3. 热门排序动态变化，跨页期间可能产生顺序变化；数据库可以去重，但不能恢复接口未返回的评论。
4. 当前只支持 BV 号，不直接接受纯 AV 号。
5. 当前只保存评论当前快照，不保存点赞数或排名的历史序列。
6. 当前无数据库迁移框架；发布修改表结构的新版本时，需要先设计显式迁移。
7. 当前无定时任务、运行指标、Web UI 或大规模任务队列，这些均不属于 v0.1.0 范围。

## 10. 后续建议

建议按实际需求逐步演进，不提前扩展：

1. 先使用少量自有测试目标观察一段时间，确认匿名接口稳定性。
2. 如确实需要讨论树，再增加楼中楼表和独立分页逻辑。
3. 如确实需要长期监控，再增加调度、快照历史和数据库迁移。
4. 如提供给非技术用户，再评估本地 Web UI，而不是直接把抓取与前端耦合。
